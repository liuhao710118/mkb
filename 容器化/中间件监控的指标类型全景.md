# 中间件监控的指标类型全景

中间件监控通常从 **四个层级** 来拆解：**资源 → 服务可用性 → 性能指标 → 业务/拓扑**。下面按「通用分类框架 + 主流中间件具体指标清单」给你一套可直接落地的体系。

------

## 一、四层通用分类框架

| 层级               | 关注什么                     | 典型指标方向                                             |
| ------------------ | ---------------------------- | -------------------------------------------------------- |
| **L1 宿主资源**    | 中间件跑在哪、吃得怎么样     | CPU、内存、磁盘 IO、网络、文件句柄、端口监听             |
| **L2 进程/运行时** | 中间件自身是否活着、是否健康 | 进程存活、JVM（如适用）/线程数、连接数、FD 数、启动时间  |
| **L3 服务性能**    | 它干活快不快、卡不卡         | 吞吐量(QPS/TPS)、延迟(P95/P99)、错误率、队列堆积、命中率 |
| **L4 业务语义**    | 对业务的影响是什么           | 消息积压条数、慢查询数、拒绝连接数、关键操作失败率       |

> 口诀：**活着（Alive）→ 够用（Resource）→ 能扛（Performance）→ 有意义（Business）**

------

## 二、按中间件类型拆解核心指标

### 1️⃣ 缓存类（Redis / Memcached）

| 类别              | 关键指标                                                     |
| ----------------- | ------------------------------------------------------------ |
| **吞吐量**        | QPS（每秒命令数）、总命令数                                  |
| **延迟**          | P95/P99 延迟、慢命令（slowlog）数量                          |
| **命中率**        | `keyspace_hits / (hits + misses)`——最经典的一个              |
| **内存**          | 使用内存、内存峰值、内存碎片率（mem_fragmentation_ratio）    |
| **连接**          | 当前连接数、rejected connections                             |
| **持久化/稳定性** | RDB/AOF last save status、fork 耗时（Redis）、主从复制延迟（offset lag） |
| **驱逐/淘汰**     | evicted_keys、expired_keys                                   |
| **错误**          | OOM、readonly 错误、主从切换事件                             |

> ⚠️ Redis 最常见告警三件套：**内存趋顶 + 命中率骤降 + eviction 飙升**

------

### 2️⃣ 消息队列（Kafka / RabbitMQ）

#### Kafka

| 类别          | 关键指标                                                     |
| ------------- | ------------------------------------------------------------ |
| **Broker**    | UnderReplicatedPartitions、OfflinePartitionsCount、ActiveControllerCount（!=1 就危险） |
| **吞吐**      | MessagesInPerSec、BytesInPerSec / BytesOutPerSec             |
| **堆积/滞后** | Consumer Lag（最关键的 SLO 指标之一）                        |
| **磁盘**      | Log dir size、磁盘使用率（Kafka 吃磁盘极凶）                 |
| **请求延迟**  | Produce/Fetch request latency（P95/P99）                     |
| **错误**      | RequestErrors、ISR shrink/expand 频次                        |

#### RabbitMQ

| 类别          | 关键指标                                                     |
| ------------- | ------------------------------------------------------------ |
| **队列**      | 队列长度（Ready + Unacked）、消息堆积字节数                  |
| **吞吐**      | Publish rate、Deliver/DeliverGet rate、Ack rate              |
| **消费者**    | 消费者数量、unack 堆积、prefetch 耗尽导致饥饿                |
| **连接/通道** | TCP connections、channels、file descriptors                  |
| **资源**      | Erlang VM memory（Rabbit 是 BEAM 虚拟机）、磁盘 alarm / mem alarm |
| **错误**      | redelivered 过多 = 消费反复失败的信号                        |

> Kafka 看 **Consumer Lag**；RabbitMQ 看 **queue depth + unacked**——本质都是"堆积"。

------

### 3️⃣ 关系型数据库（MySQL / PostgreSQL）

| 类别            | 关键指标                                                     |
| --------------- | ------------------------------------------------------------ |
| **连接**        | 活跃连接数、threads_connected、max_connections 使用率、aborted connections |
| **查询性能**    | QPS、TPS、慢查询数（slow_queries/sec）、平均查询耗时         |
| **锁**          | InnoDB row lock time/wait（MySQL）、deadlocks、waiting queries |
| **缓冲池**      | Buffer Pool hit rate（MySQL）/ shared_buffers cache hit（PG）——命中率低=P1级 |
| **事务/写压力** | 并发事务数、binlog/redo write throughput、WAL 延迟（PG）     |
| **复制**        | 主从延迟（Seconds_Behind… / replication_lag）、IO/SQL thread 状态 |
| **错误**        | 死锁次数、连接被打断数、临时表落盘、disk full                |

> DB 监控的灵魂两句话：**"有没有慢查询泛滥？"** 和 **"缓冲池还热不热？"**

------

### 4️⃣ NoSQL / 文档库（MongoDB / ElasticSearch）

#### MongoDB

- 连接数、opcounters（insert/update/query/delete）、scanned objects vs returned ratio

- Page faults（内存不够的迹象）、replication lag、index usage / missing index 查询

- WT cache dirty% / used%（WiredTiger cache 是命门）

#### Elasticsearch

- 索引速率 / 搜索速率（indexing rate / search rate）

- 集群健康：`status`（green/yellow/red）、unassigned shards

- JVM heap 使用率（ES 对 heap 极其敏感，>85% 就很危险）

- 刷新/合并延迟、bulk reject、circuit breaker tripped

- disk watermark（flood-stage 会直接拒写）

------

### 5️⃣ Web / 代理层（Nginx / HAProxy / Envoy）

| 类别         | 关键指标                                         |
| ------------ | ------------------------------------------------ |
| **流量**     | 请求数/sec、带宽 in/out                          |
| **延迟**     | upstream_response_time P95/P99、request_time     |
| **错误**     | 4xx/sec、5xx/sec、upstream 超时/不可用次数       |
| **连接**     | active connections、waiting、accepted vs handled |
| **上游健康** | upstream fail count、健康检查结果                |

------

### 6️⃣ JVM 系中间件（Tomcat / Spring Boot / RocketMQ / ES…）

如果是 JVM 中间件，额外盯住：

- **Heap Used / Max、GC Pause（尤其 Full GC 频次和耗时）**

- Thread count、deadlocked threads

- Metaspace、direct buffer memory

- （很多"中间件变慢"的根因其实是 GC 抖动）

------

## 三、你真正该建的监控模型（落地建议）

### 1）黄金信号（最适合做 SLO/告警）

| 信号                   | 对应指标                                                     |
| ---------------------- | ------------------------------------------------------------ |
| **Traffic**            | QPS / msg/s / bytes/s                                        |
| **Latency**            | P95/P99（别只看 avg）                                        |
| **Errors**             | 4xx/5xx、写入失败、拒绝、dead letter                         |
| **Saturation（饱和）** | 连接数使用率、内存使用率、队列深度/consumer lag、线程池活跃度 |

### 2）告警优先级模板

- 🔴 **P0**：服务不可达 / 主从全挂 / 磁盘满 / 拒绝写入

- 🟠 **P1**：堆积持续上涨 / 慢查询暴增 / 命中率骤降 / consumer lag > 阈值且增长

- 🟡 **P2**：内存趋势上涨但未触顶 / FD 偏高 / replication lag 偶发抖动
