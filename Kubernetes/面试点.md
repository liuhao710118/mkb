# service是命令空间级别的资源

**Service 是命名空间级别（namespace-scoped）资源，不是集群级别（cluster-scoped）的。**

------

## 最直接的证据

```
kubectl api-resources | grep service
# NAME      SHORTNAMES   APIVERSION   NAMESPACED   KIND
# services  svc          v1           true         Service   ← NAMESPACED=true
```

这意味着：

- Service **必须属于某个 namespace**，创建时必须指定（不指定就默认 `default`）

- `kubectl get svc -A`实际上是去 **每个 namespace 分别列**，而不是从一个全局表里取

- 不能用 `kubectl get svc --all-namespaces`之外的"全局 Service"概念

------

## 最容易混淆的点：它"活在 namespace 里"，但 ClusterIP 是"全集群可达"

这一点是所有人第一次卡住的地方：

|                          | 说明                                                         |
| ------------------------ | ------------------------------------------------------------ |
| **Service 对象本身**     | 在 `default`/ `prod`/ `kube-system`等某个 namespace 里 ✅ namespace-scoped |
| **Service 的 ClusterIP** | 是虚拟 IP，但 **集群内任意节点/Pod 都能路由到它**（由 kube-proxy 保证）✅ cluster-wide reachable |
| **Service 的 DNS 名**    | 默认只解析短名 `my-svc`在同 namespace，全名必须带 namespace：`my-svc.other-ns.svc.cluster.local` |

所以：

```
# 在 ns-a 的 Pod 里访问 ns-b 的 Service
curl http://order-svc.ns-b.svc.cluster.local:8080
```

> **Service 是 namespace 的"居民"，但它对外暴露的 ClusterIP 是整个集群网络的公共基础设施。**

------

## 一句话总结

> **Service 的「对象归属」是 namespace 级的；Service 的「网络效果」是 cluster 级的。**
>
> 它不是 cluster-scoped 资源（像 Node、PV、ClusterRole 那样），别被 ClusterIP 那个"Cluster"前缀误导。

如果你接下来在看 **跨 namespace 调用怎么管**（能不能限制谁可以调谁），那就自然落到 **NetworkPolicy** 或 service mesh 的领地了。



# service的四种类型

| 类型 (Type)          | 核心作用                        | 是否有 ClusterIP | 对外暴露方式                                                 | 典型使用场景                                                 | 关键注意点                                                   |
| -------------------- | ------------------------------- | ---------------- | ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **ClusterIP** (默认) | **集群内部通信**                | ✅ 有 (虚拟 IP)   | **不对外暴露**。 仅在集群内部通过 `ClusterIP:Port`访问。     | 后端微服务之间的调用（如：订单服务调用库存服务）。           | 这是最常见的类型。如果不指定 `type`，默认就是它。            |
| **NodePort**         | **早期简易对外暴露**            | ✅ 有             | **在所有 Node 的 IP 上**开放一个**静态端口** (30000-32767)。 访问方式：`任何NodeIP:NodePort`。 | 临时测试、简单的对外暴露、或在没有云 LB 的环境下手动搭建入口。 | 端口范围受限且不直观；客户端需要知道节点 IP；节点下线会导致访问失败。 |
| **LoadBalancer**     | **生产级标准对外暴露** (云环境) | ✅ 有             | **云厂商提供的负载均衡器 (LB)**。 提供一个**稳定的外部 IP** (或 DNS) 和标准端口 (如 80/443)。 | 生产环境中，需要对外提供稳定入口的 Web 服务或 API。          | **依赖云厂商**。在裸金属环境下需要 MetalLB 等插件支持。底层通常仍借用 NodePort。 |
| **ExternalName**     | **DNS 别名 (映射外部服务)**     | ❌ 无 (只有 DNS)  | **不做端口转发**，仅通过 DNS CNAME 记录将服务名指向一个**外部域名**。 | 将集群外的服务（如 AWS RDS、第三方 API）伪装成集群内部服务，方便迁移或解耦。 | **只能是域名**，不能是 IP；访问 HTTPS 时极易遇到 **TLS 证书校验失败**（SNI/Host 不匹配）问题。 |

------

### 💡 补充：如何选择？

为了方便你决策，这里有一个简单的**决策树**：

1. **只给集群内部用？**
   -    **是** → 用 `ClusterIP`。
   
2. **要给外面用？**
   -    **是，且是生产环境/云上** → 用 `LoadBalancer`（最省心，稳定入口）。
   
   -    **是，但只是临时测试/裸机没LB** → 用 `NodePort`（凑合用，记得防火墙放行）。
   
3. **服务其实在集群外面（比如老数据库）？**

   -    **是，且只有域名** → 用 `ExternalName`（注意证书问题）。

   -    **是，且有固定 IP** → 用 `ClusterIP`+ **手动 EndpointSlice**（比 ExternalName 更可控）。



# service中loadbalancer解决了什么问题

> LoadBalancer 解决的核心问题：**把"对外暴露服务"从 Node 层的粗糙方案，提升到面向外部世界的可用入口**

### 先承认 NodePort 能工作，但它解决的是"能通"，不是"能用"

NodePort 做到了一件事：**让集群外的流量进得来**。但付出的代价是强迫外部世界接受三个不合理的事实：

1. **入口绑死在某台 Node 的 IP 上**

   -    节点重建 / 缩容 / IP 变更 → 入口跟着挂

   -    外部要自己处理"哪个 Node 还活着"——这本该是集群内部的事

2. **端口是 30000–32767 这种不可辨识、不可发布的端口**
   -    你没法跟别人说"访问我们的 API 用 `:30456`"
   
   -    用户期待的是 `:80`/ `:443`，不是机器后门号
   
3. **安全策略被迫铺到每一台 Node 上**

   -    防火墙 / 安全组要对所有 Node 放行那个高位端口

   -    攻击面=Node数量，Node越多越难管

所以 NodePort 的本质问题是：**它暴露的是"集群内部的机器"，而不是"一个服务"。**

------

### LoadBalancer 的答案：插一个独立于 Node 的存在，对外只给一个稳定面孔

```
外部世界  →  [ 固定 IP / DNS : 80|443 ]  ← 这就是 LB
                  ↓ 转发
           健康的 Node : NodePort（内部细节，外部不必知道）
                  ↓ kube-proxy
               Pod
```

它一次性解决三件事：

| 问题           | LoadBalancer 的做法                                  |
| -------------- | ---------------------------------------------------- |
| **入口不稳定** | 入口 IP/DNS 属于 LB 本身，跟任何 Node 的生命周期无关 |
| **端口不标准** | 对外可以挂 80/443，外部只看得到 LB 的端口            |
| **安全面失控** | 防火墙/安全组只管 LB 这一层；Node 不需要对公网开洞   |

外加两个 NodePort **做不到**但生产必需的附加值：

- **健康检查**：LB 知道哪些 Node 还能服务，只把流量发给活的（NodePort 本身不管这个）

- **四层卸载点**：很多云 LB 还能顺带做来源 IP 保留（`externalTrafficPolicy: Local`的取舍）、ACL、流日志等

------

### 一句话定义它"解决的问题域"

> NodePort 回答的是：**"怎么让外面的包穿透到集群里？"**
>
> LoadBalancer 回答的是：**"外面世界需要一个什么样的地址来访问这个服务——而且要像一个正常的、稳定的、可运维的网络服务？"**

前者是连通性问题，后者是**接口契约问题**。

------

### 最后补一刀：它为什么不是终点，而是中间站

LoadBalancer 解决完"稳定入口"后，马上会引出下一个问题：

> **我有 50 个服务，难道买 50 个 LB？**

这才是 Ingress（七层，一个 LB 后面按域名/路径收拢 N 个服务）出现的动机。所以完整链条是：

```
ClusterIP  →  内部能通
NodePort    →  外部能通（但丑、脆、难管）
LoadBalancer → 外部能通 + 稳定入口 + 可运维
Ingress     → 多个服务共享同一个 LB，按 HTTP 语义分流
```

**LoadBalancer 在整个链条里的定位就是：把 NodePort 的"机器级入口"升级成"服务级入口"。** 这件事它解决得很干净，但它本身不负责"多个服务的路由整合"——那是隔壁 Ingress 的活。

# service中externalName解决了什么问题

## `type: ExternalName`解决的到底是什么问题？

一句话先定性：**它不是为了解决“流量怎么进来/出去”而生的，而是为了解决「集群内部对一个外部系统的“名字与依赖抽象”」而生的。**

> 它做的是 **DNS 层的别名（CNAME）**，把
>
> **`某个内部服务名（通常在 \*.svc.cluster.local）`**
>
> 变成
>
> **`一个外部域名`**
>
> 从而让 Pod 里的一切继续用“调服务”的思维方式，而不是到处硬编码外部域名。

------

## 具体解决的三件痛点

### 1) 把「外部依赖地址」从应用代码/配置里抬出来

没有它时，你的服务可能是这样的（散落在各处）：

- `DB_HOST=billing.example.net`

- `HTTP_CLIENT_BASE=https://billing.example.net`

这意味着：

- 外部域名是“谁家、会不会变、迁不迁、切不切”你得去改应用配置/镜像

- 有一天你想把外部系统迁进集群，就要做一次大规模“去外部域名化”改造

用了 ExternalName：

```
# 内部统一叫 legacy-billing
name: legacy-billing
type: ExternalName
externalName: billing.example.net
```

应用代码改成：

- `legacy-billing`（DNS 解析走 CNAME → billing.example.net）

效果是：**外部系统的“位置”被集中收口到一个 K8s 对象上**，改一个 Service 就能改寻址（至少在名字层）。

> 这在迁移期尤其值钱：你先“挂羊头卖狗肉”（名字指向外部），后面再把 Service 换成真正 in-cluster 的 ClusterIP + selector/Pod，**调用方尽量零感”。

------

### 2) 让外部系统也能“混入”集群的服务寻址习惯（统一心智模型）

集群内部天然的语言是：

```
<service>.<namespace>.svc.cluster.local
```

ExternalName 让你可以写：

- `legacy-billing.prod.svc.cluster.local`

而不是让团队一半人写 `localhost:3306`、一半写 `10.0.3.7:3306`、一半写 `billing.example.net`。

它解决的是：**不要让“它在外面”破坏你们在集群里的命名一致性。**

------

### 3) 让“服务目录/依赖图”至少在名字层面完整（可文档化、可治理）

当你们有服务树、依赖清单、或后续想上工具（拓扑图、策略、审计）时：

- 一个 `Service`对象至少能告诉你：**“我这个名字代表一个依赖”**

- 否则外部依赖对集群来说就是“黑盒字符串”，很难纳入统一治理口径

------

## 它**明确不解决**什么（这点更重要）

| 你觉得它可能解决                 | 真相                                                         |
| -------------------------------- | ------------------------------------------------------------ |
| 提供 ClusterIP / 四层转发        | ❌ 没有 ClusterIP，不生成 kube-proxy 规则，本质是 DNS         |
| 帮你规避外部 HTTPS 证书/SNI 问题 | ❌ CNAME 不改变 TLS 握手中的 SNI/Host（所以我们前面才引入反代） |
| 把“外部 IP”塞进来                | ❌ `externalName`必须是域名，不是 IP                          |
| 给流量做健康检查/负载均衡/可观测 | ❌ 流量是 Pod 直连外部域名；集群只看 DNS 结果                 |

所以它解决的是 **naming/indirection（命名间接层）**，不是 **routing/proxying（路由/转发）**。

------

## 一句话收口

> **ExternalName 解决的是：当你暂时必须把流量发到集群外的一个域名时，至少让集群内部用一个“正当的 Service 名”去引用它，避免让外部域名侵入应用配置，并为将来把它们“收回集群”留出低成本切换点。**
>
> 但它不替你处理证书、不给你 ClusterIP、也不帮你管控出口——那些要靠反代（Nginx/Envoy）或 EndpointSlice 这类更重的方案。

如果你愿意，你把你现在这个外部系统描述成一句：**“协议 / 域名还是IP / 能不能动证书 / 是否是迁移过渡期”**，我可以帮你拍板：**ExternalName 适不适合你现在用**，还是该一步到位走反代/EPSlice。

# service中的externalName存在证书验证问题

> externalName虽然可以给外部系统提供对k8s集群内的抽象访问 看起来像是在一个集群内的服务一样 但是本质上就是对外部系统的域名做了CNAME(别名) 对于无证书验证的http服务没有什么大问题 但是对于有证书验证的https服务 就会导致证书验证失败 因为内部访问是用的内部虚拟出来的域名 而证书里面能验证的域名是实际有效的域名
>
> 就比如 内部你把bilibi.com抽象了 内部可以直接通过https://bilibi-service访问 这样是能解析到IP的 但是证书验证是过不去的 证书只能验证https://bilibi.com



# service中的externalName证书问题如何解决

> 推荐是在集群内部启动一个 nginx 反向代理Pod 内部系统通过访问nginx代理 再由代理去访问外部系统 同样能达到抽象外部系统的目的 且 证书问题也完美解决了

# service中的endpoint与endpointSlice

> 旧版本是一个service对于一个endpoint 但是目前的新版本则是 一个service对应0-N个endpointslice 每100个IP作为一个endpointslice集合 超过则自动新建第二个

### 一句话先定性

> **Endpoints 是旧时代（v1 core）的"单对象存所有后端 IP"；EndpointSlice 是它的替代方案（discovery.k8s.io/v1，GA v1.21），把后端拆成多片小对象。Endpoints 在 v1.33 起已正式 deprecated。**

------

### 核心对比表

|                | **Endpoints**（旧）                                          | **EndpointSlice**（新）                                      |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **API**        | `v1`（core API）                                             | `discovery.k8s.io/v1`                                        |
| **归属**       | 命名空间级，与 Service **同名且 1:1**（一个 SVC 只有一个 Endpoints） | 命名空间级，一个 SVC 对应 **0~N 个** EndpointSlice           |
| **结构**       | 一个大对象，所有 endpoint 塞进 `subsets[].addresses[]`       | 小对象，每片最多 ~100 个 endpoint（可调），按 **IP族/端口集** 拆分 |
| **地址类型**   | IPv4 仅（主地址族）                                          | **IPv4 / IPv6 / FQDN**，双栈时拆两片                         |
| **拓扑信息**   | 几乎没有（只有 `addresses[].nodeName`这种边角）              | 正式字段：`nodeName`、`zone`、`conditions`（ready / serving / terminating） |
| **更新效率**   | Pod 增删 → **整个对象替换** → 大 watch event 广播给所有观察者（kube-proxy 等） | Pod 增删 → **只更新受影响的那个 Slice** → watch event 很小   |
| **规模上限**   | 单对象无限膨胀（上千 Pod 时巨大 JSON）                       | 天然分片，默认 ≤100/ep 每片，可控可预测                      |
| **新特性承载** | ❌ 不再加新功能（双栈、traffic distribution 等新功能只走 EndpointSlice） | ✅ 所有新 Service 网络特性都在这里                            |
| **当前状态**   | **Deprecated since v1.33**（会打印 warning，短期不会删但别再写新代码用它） | **当前唯一正确的消费/生成目标**                              |

------

### 它们的位置关系（最关键）

```
Service（有 selector）
    │
    │  endpoints 控制器 / endpointslice 控制器  watch Pod 变化
    ▼
┌───────────────────────┐        ┌─────────────────────────────────┐
│  Endpoints（旧，兼容残留）│   ←自动镜像─→  │  EndpointSlice(s)（新，真相源）  │
│  myservice             │        │  myservice-7vzhx (IPv4 切片)     │
│  subsets[].addresses   │        │  myservice-jcv8s (IPv6 切片)     │
└───────────────────────┘        └─────────────────────────────────┘
                                           │
                                      kube-proxy 读的是它（现代集群默认）
```

注意：**真正干活的是 EndpointSlice**。在 v1.22+ 的 kube-proxy 默认配置下，流量路由规则基于 EndpointSlice 计算；Endpoints 对象更多是"历史遗留的镜像/兼容层"，方便老脚本还能 `kubectl get endpoints`看到东西。

来验证一下你日常能看到的现象：

```
# 旧视角（还能看，但会看到 deprecation warning ≥1.33）
kubectl get endpoints my-svc
# NAME      ENDPOINTS                     AGE
# my-svc    10.244.1.5:80,10.244.2.3:80   5m

# 新视角（推荐）
kubectl get endpointslice -l kubernetes.io/service-name=my-svc
# NAME               ADDRESSTYPE   PORTS   ENDPOINTS
# my-svc-abc123      IPv4          80      10.244.1.5,10.244.2.3
```

------

### 为什么 Endpoints 非要被换掉 —— 一个规模问题

假设一个 Service 后面有 **2000 个 Pod**：

|                    | Endpoints                                                | EndpointSlice                               |
| ------------------ | -------------------------------------------------------- | ------------------------------------------- |
| **对象数**         | 1 个，体积几十~上百 KB 的 JSON                           | ~20 个 Slice，每片 ~100 endpoint，每片几 KB |
| **加/减 1 个 Pod** | 序列化 **整份列表** → 写 API Server → 所有人收到整份变更 | 只改 **1 个 Slice** → 小更新传播            |
| **watch 风暴**     | 每次滚动发布 = 全量事件砸给每个 kube-proxy               | 增量、局部、可控                            |

当集群一大，这个差异直接从"不优雅"变成"API Server 压力/节点 CPU 浪费"。

------

### 实操上你该怎么对待它们

| 场景                                     | 该用哪个                                                 | 说明                                                         |
| ---------------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------ |
| **你只是看问题**（`kubectl get ep`排障） | 看 Endpoints 也行（直观），但养成看 EndpointSlice 的习惯 | `kubectl get endpointslice -l k8s.io/service-name=xxx`       |
| **你写 YAML/控制器，手动托管外部 IP**    | ✅ **手写 EndpointSlice**（discovery.k8s.io/v1）          | 就是我们前面做的那个外部 IP 例子                             |
| **你写 YAML/控制器，手写旧的 Endpoints** | ❌ 别新写                                                 | 官方明确：无 selector 的 Service 要手动指定端点 → **直接写 EndpointSlice** |
| **你读 API/Watch 做服务发现（程序里）**  | ✅ 读 EndpointSlice List + LabelSelector                  | 不再是按名字 Get，而是按 `kubernetes.io/service-name=xxx`过滤 |

------

### 你前面手动写外部 IP 的例子，用 Endpoints 的旧写法长这样（⚠️ 不推荐新代码）

```
# ⚠️ 旧写法 —— 能跑但属于 deprecated 路径
apiVersion: v1
kind: Endpoints
metadata:
  name: ext-mysql   # 必须和 无selector 的 Service 同名
subsets:
- addresses:
  - ip: 203.0.113.10
  ports:
  - port: 3306
```

换成我们之前给你的 **EndpointSlice 写法**就是当前正确姿势 —— 不需要担心 deprecation warning，也不需要依赖镜像兼容链。

------

### 一句话收口

> **Endpoints = 单体大胖对象，1 Service 绑 1 个，因规模化瓶颈被淘汰；EndpointSlice = 分片小对象，1 Service 对 N 片，是当前及未来的唯一真相源。你手动接外部 IP 时直接写 EndpointSlice 就对了。**



# ingress是命名空间级别的资源

## Ingress 是 **命名空间级别（Namespaced）** 的资源

### 直接结论

```
kubectl api-resources | grep ingress
# NAME      SHORTNAMES   APIVERSION                     NAMESPACED   KIND
# ingresses   ing         networking.k8s.io/v1           true         Ingress
```

`NAMESPACED = true`，所以 **Ingress 属于某个具体 Namespace**，不能脱离 Namespace 存在。

------

### 这意味着什么？

| 特性                                  | 说明                                                         |
| ------------------------------------- | ------------------------------------------------------------ |
| **创建时必须指定 namespace**          | `kubectl apply -n my-app -f ingress.yaml`                    |
| **只能路由到同 namespace 的 Service** | Ingress 的 `backend.service.name`引用的 Service **必须在同一个 namespace** 内 |
| **同名没关系，不同 ns 互不影响**      | `default/my-ingress`和 `team-a/my-ingress`是两回事           |
| **`kubectl get ingress -A`**          | 跨所有 namespace 列出，但不改变它是 namespaced 的事实        |

------

### 那为什么 Ingress *看起来*"全局"？

因为真正"干活"的东西在 namespace 之外：

1. **Ingress Controller** 本身是集群级的 —— 它通常以 DaemonSet / Deployment 形式跑在专属 namespace（如 `ingress-nginx`），**Watch 所有 namespace 的 Ingress 对象**。

2. 所以它把所有 namespace 里的 Ingress 规则汇总起来，统一写到一台/多台 NGINX（或其他 LB）配置里，对外呈现为一个统一的入口点。

> 换句话说：**规则是分 namespace 隔离的，但汇聚后的生效面是集群级。** 这是"看起来全局"的根源。

------

### Ingress vs IngressClass（容易混的一点）

| 资源             | 级别             | 作用                                                         |
| ---------------- | ---------------- | ------------------------------------------------------------ |
| **Ingress**      | ✅ Namespaced     | 描述"某 namespace 的流量路由规则"                            |
| **IngressClass** | ❌ Cluster-scoped | 定义"用哪套 Ingress Controller"（集群唯一），通过 `spec.ingressClassName`或老式 `kubernetes.io/ingress.class`annotation 与 Ingress 关联 |

```
# Ingress — 必须放在某个 namespace
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: foo
  namespace: app-ns
spec:
  ingressClassName: nginx          # 指向集群级的 IngressClass
  rules:
  - host: foo.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: foo-svc           # ← 这个 Service 也必须在 app-ns 里
            port:
              number: 80
```

------

### 一句话总结

> **Ingress 是 Namespaced 资源，规则天然隔离在各 namespace 内；Ingress Controller 是集群级的消费者，负责把这些分散的规则合并成一份对外可达的路由配置。** 如果你需要做"跨 namespace 的 Service 路由"，通常做法是：让目标 Service 暴露出来，或在目标 namespace 也建对应的 Ingress，或通过两层 Ingress/Gateway API 来解决。
